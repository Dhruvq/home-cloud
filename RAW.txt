Part 1:

1. The Core Architecture
You are replacing Windows with a professional-grade server operating system to avoid forced reboots and maximize hardware power.
The OS: Proxmox VE (A Type-1 Hypervisor). It runs "bare metal" on the PCs.
The Network: Tailscale. This acts as a magic, secure tunnel connecting your Mac to the remote machines from anywhere in the world, without touching router port forwarding.
The Workloads: You will run your projects (like OpenClaw) inside lightweight Linux Containers (LXC) or Virtual Machines (VMs) on Proxmox.
2. Remote Location "Golden Rules"
Since these machines will live in a different location from you, you must follow these rules:
The "Same Roof" Rule: All three PCs must be physically plugged into the same local network (router/switch) at the remote location so they can talk to each other without internet lag.
Power Outage Proofing: You must go into each PC's BIOS and set "Restore on AC Power Loss" to "Power On" so they boot themselves back up if the remote location loses electricity.
3. The Installation (The flash drive route)
You are using your portable flash drive as the temporary "delivery truck" to install Proxmox onto the PC's internal drive.
CRITICAL WIPE WARNING: > 1. Flashing the installer will completely wipe everything currently on your portable flash drive.
2. Installing Proxmox will completely wipe Windows and all files on the PC's internal drive.
The 4-Step Process:
Use your Mac and a program like BalenaEtcher to flash the Proxmox ISO onto your portable SSD.
Plug the SSD into the Windows PC and boot from it (via BIOS settings).
Follow the wizard to install Proxmox onto the PC's internal drive.
Unplug the portable SSD, let the PC reboot, and you are done with the SSD (you can reformat it for your Mac later).
4. The Rollout Strategy
Do not set this up for the first time at the remote location.
Phase 1: Wipe one PC at home next to your Mac. Install Proxmox, set up Tailscale, and make sure you can log into the dashboard via your Mac's browser.
Phase 2: Move that one working PC to the remote location and verify it comes back online.
Phase 3: Wipe the other two PCs, move them, and join them to the first one to create your "Supermachine" cluster

Enabling SSH Access to Proxmox Ubuntu VMs Using Tailscale
Overview
This document explains how to enable secure SSH access to Ubuntu virtual machines using Tailscale.
Each VM must authenticate with Tailscale individually to obtain a Tailscale network IP (100.x.x.x).
Once authenticated, the VM can be accessed remotely via SSH.

Prerequisites (for PREREQ)
Ubuntu VM running
Tailscale installed on the VM
Access to the VM console through Proxmox
Tailscale account (Google login in this case)
Nomad CLI

Step 1 — Verify Tailscale Status
Open the VM console in Proxmox and run:
tailscale status
If Tailscale is not connected, you will see a message indicating the node is logged out or inactive.

Step 2 — Authenticate the VM with Tailscale
Run:
sudo tailscale up
The command will output a login URL similar to:
https://login.tailscale.com/a/XXXXXXXX
Open this link in your browser and authenticate using the same account used for your Tailscale network (Google login).
After authentication, the VM will join your Tailnet.

Step 3 — Confirm Tailscale IP Address
Run:
tailscale ip
You should receive an IP address in the following format:
100.x.x.x
This is the secure internal Tailscale IP assigned to the VM.

Step 4 — Enable SSH Password Authentication
Edit the SSH configuration:
sudo nano /etc/ssh/sshd_config
Find the following line:
PasswordAuthentication
Ensure it is set to:
PasswordAuthentication yes
Save the file.
Restart the SSH service:
sudo systemctl restart ssh

Step 5 — Connect to the VM via SSH
From your local machine, run:
ssh ubuntu@100.x.x.x
Enter the Ubuntu user password when prompted.
Note: The terminal will not display characters while typing the password.

Step 6 — Enable Tailscale on Boot
To ensure Tailscale starts automatically after reboot:
sudo systemctl enable tailscaled

Step 7 — Repeat for Additional Nodes
Each VM must authenticate separately.
Repeat Steps 1–6 for:
Node 1
Node 2
Node 3
After completing this process, all nodes will appear in the Tailscale admin console and can securely communicate with each other.

Verification
You can verify connectivity by SSHing into each node using its Tailscale IP:
ssh ubuntu@100.x.x.x
If successful, you will receive the Ubuntu shell prompt.

Notes
• Each VM is treated as a separate device in Tailscale.
• Authentication is only required once per VM.
• Tailscale provides secure networking without exposing the VM to the public internet.




HashiCorp Nomad
Nomad is like a much simpler, more flexible version of Kubernetes. It is very popular for small home labs.
How it works: It’s a single binary you run on all nodes. It handles "Job Scheduling" and natively understands NVIDIA GPUs very well.
Difficulty: 6/10. It’s easier to set up than Kubernetes but more robust than a custom Python script.
The Reward: Great balance of "set it and forget it" without the massive overhead of K8s
Step 1: Install Nomad on all 3 Nodes
You need to run these commands on the Proxmox Shell of Node 1, Node 2, and Node 3. Nomad is just a single binary, so the installation is identical for all.
Add the HashiCorp GPG key:
Bash
wget -O- https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
Add the official repository:
Bash
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
Install Nomad:
Bash
sudo apt update && sudo apt install nomad -y

Step 2: Configure Node 1 as the "Server+Client" (The Leader)
Your laptop will manage the schedule. It won't run the heavy AI jobs itself; it just tells the desktops what to do.
Edit the config: sudo nano /etc/nomad.d/nomad.hcl
Replace the contents with this:
Terraform
data_dir  = "/opt/nomad/data"
bind_addr = "0.0.0.0" # Listen on all interfaces

server {
  enabled          = true
  bootstrap_expect = 1 # We only need 1 leader for this small cluster
}
client {
  enabled = true
}
plugin "docker" {
  config {
    allow_privileged = true
    allow_caps = ["ALL"]
  }
}
advertise {
  # Use your Node 1 Tailscale IP here
  http = "100.x.x.x"
  rpc  = "100.x.x.x"
  serf = "100.x.x.x"
}


Start it up:
Bash
sudo systemctl enable nomad
sudo systemctl start nomad



Step 3: Configure Node 2 & 3 as "Clients" (The Muscle)
These machines will actually talk to the GPUs.
Edit the config on Node 2 (and later Node 3): sudo nano /etc/nomad.d/nomad.hcl
Use this Client config:
Terraform
data_dir = "/opt/nomad/data"
bind_addr = "0.0.0.0"

client {
  enabled = true
  # Point this to Node 1's Tailscale IP
  servers = ["100.NODE_1_IP:4647"]
}

plugin "docker" {
  config {
    allow_privileged = true
    allow_caps = ["ALL"]
  }
}

advertise {
  # Use the Tailscale IP of the CURRENT node (Node 2 or 3)
  http = "100.x.x.x"
  rpc  = "100.x.x.x"
  serf = "100.x.x.x"
}


Start it up: sudo systemctl enable nomad && sudo systemctl start nomad

Step 4: The Moment of Truth
Go back to your Mac terminal and type: nomad node status -address=http://100.NODE_1_IP:4646
If you see all three nodes listed with a status of "ready", your auto-scheduler is live.

Step 5: Install the NVIDIA Device Plugin
Nomad does not detect GPUs out of the box. You need to install a separate plugin binary on every node that tells Nomad's scheduler about the available GPUs. Without this, Nomad has no idea GPUs exist even if the NVIDIA drivers and Docker runtime are working perfectly.
Run the following on all 3 nodes:
Create the plugin directory:
bash
sudo mkdir -p /opt/nomad/plugins
Download and install the plugin binary:
bash
wget https://releases.hashicorp.com/nomad-device-nvidia/1.1.0/nomad-device-nvidia_1.1.0_linux_amd64.zip
unzip nomad-device-nvidia_1.1.0_linux_amd64.zip
sudo mv nomad-device-nvidia /opt/nomad/plugins/
sudo chmod +x /opt/nomad/plugins/nomad-device-nvidia
rm nomad-device-nvidia_1.1.0_linux_amd64.zip
Now update /etc/nomad.d/nomad.hcl on each node to tell Nomad where the plugin lives and to enable it. Add these two blocks to the existing config:
hcl
plugin_dir = "/opt/nomad/plugins"

plugin "nomad-device-nvidia" {
  config {
    enabled            = true
    fingerprint_period = "1m"
  }
}
Restart Nomad on each node after saving:
bash
sudo systemctl restart nomad
To confirm the plugin loaded correctly, check the logs on any node:
bash
sudo journalctl -u nomad | grep -i nvidia
```

You should see a line like:
```
agent: detected plugin: name=nvidia-gpu type=device plugin_version=1.1.0
Verify GPU Registration from your Mac:
Run this for each node ID:
bash
nomad node status -verbose <node-id>
```

You are looking for a **Device Group Capacities** section that was not there before:
```
Device Group Capacities
Device Group                          Count  Healthy  Unhealthy
nvidia/gpu/NVIDIA GeForce RTX 3080    1      1        0
All three nodes should report their respective GPUs. At this point Nomad's scheduler is fully GPU-aware and will only place GPU jobs on nodes that have an available device.

Step 6: Run Your First GPU Job
Now that the scheduler knows about your GPUs, you need to request them correctly in your job file. The key difference from a normal job is the device stanza inside resources. This is what tells Nomad to reserve a GPU for the task and pass it through to the Docker container automatically.
Create a file called gpu-test.nomad on your Mac:
hcl
job "gpu-test" {
  datacenters = ["dc1"]
  type        = "batch"

  group "gpu-group" {
    task "cuda-check" {
      driver = "docker"

      config {
        image   = "nvidia/cuda:12.4.1-base-ubuntu22.04"
        command = "nvidia-smi"
      }

      resources {
        cpu    = 500
        memory = 512

        device "nvidia/gpu" {
          count = 1
        }
      }
    }
  }
}
Run it:
bash
nomad job run gpu-test.nomad
Grab the allocation ID from the output and check the result:
bash
nomad alloc logs <alloc-id>
A successful run will print the full nvidia-smi table showing the GPU name, driver version, CUDA version, and memory. The task will show as dead with Exit Code: 0 — this is expected and correct for batch jobs in Nomad. Dead means finished, not failed.
Important distinctions to remember:
type = "batch" is for one-shot tasks like this test. The task goes dead when it completes successfully.
type = "service" is what you will use for long-running workloads like inference servers. Those stay running.
Never use device_requests inside the Docker config block — that is a raw Docker API field that Nomad ignores entirely. Always use the device stanza inside resources.
